### Fixed
- **Subagents now load their domain skill via explicit Read-tool call** (T-1014): Fixes a framework regression where every specialist subagent (`backend`, `frontend`, `data-engineer`, `qa`, `security`, `devops`, `code-review`) spawned from the orchestrator silently bypassed its domain skill — `skills/{role}/SKILL.md` was never read into the subagent context, so the `⚡ {Role} joined` opening line, the role's anti-pattern catalog, and the per-role Output Signature blocks (Endpoint-Spec for backend, Component-Spec for frontend, AC-Pass/Fail-Tabelle for qa, Build-Report for devops) all silently dropped out of subagent output. Root cause: subagents spawned via the `Agent` tool inherit only their `agents/{name}.md` definition (identity + tool whitelist) and have no `Skill` tool in their allowed-tools list, so the previous "Lade dein Domain-Skill via Skill-Tool" instructions in agent definitions were unreachable. The minimal fix re-uses the `Read` tool that every specialist already has: every agent definition now opens its workflow with `### 1. Domain-Skill laden — ERSTER TOOL-CALL DIESER SESSION` and an explicit `Read('skills/{role}/SKILL.md')` instruction that runs before any other action; `agents/orchestrator.md` Phase 2 prompt-template and the per-phase spawn templates in `commands/develop.md` (Phase 5 implementation, Phase 6.5 code review, Phase 7 QA) now require every spawn-prompt to start with the same `Read('skills/{role}/SKILL.md')` line, with a complete role→skill-path mapping table embedded in the orchestrator template (`backend → skills/backend/SKILL.md`, `data-engineer → skills/data-engineer/SKILL.md`, `frontend → skills/frontend-design/SKILL.md` plus `creative-design` for greenfield, `qa → skills/webapp-testing/SKILL.md` plus `test-driven-development` for bugfixes; `security`, `devops`, `code-review` have no local domain skill yet and read `CLAUDE.md` + `project.json` instead). The orchestrator's `## Skill-Loading` section gained an explicit split between "for yourself (you have the Skill tool)" and "for spawned subagents (no Skill tool — must use Read instruction in spawn-prompt)" so future maintainers don't reintroduce the bug by adding a "use the Skill tool" instruction to subagent prompts. Frontend agent's stale `skills:` frontmatter cleaned up (`design` removed since the skill no longer exists; `frontend-design`, `creative-design`, `design-lead` retained as the actual targets) and one stale `(design + frontend-design Skills)` reference in `agents/orchestrator.md` Design-Modus block fixed to `(frontend-design Skill)`. Triage agent (`tools:` empty, no Read available, no domain skill) is intentionally not part of the change — its identity already lives entirely in the agent definition. Out of scope per ticket and explicitly deferred to a follow-up architecture ticket: extending `pipeline/lib/load-skills.ts` to auto-inject skills into subagent spawns via SDK appendSystemPrompt mechanism (requires SDK research that wasn't feasible in this scope), and a PreToolUse hook that would enforce the first-tool-call-must-be-skill-read invariant at the runtime layer. Reproduction reference: T-1009 backend-subagent (agentId `a46fe78e7a5532e5a`, 71188 tokens, 25 tool-calls — `skills/backend/SKILL.md` never read).

### fix: Bash permission syntax (`bb072e7`)
- `Bash(**)` is **invalid** — double-star only works for file-based tools (Read, Edit, Glob, Grep)
- `Bash(*)` or `Bash` (no parens) is the correct syntax to allow all commands
- Key gotcha: Claude Code parses shell operators (`&&`, `|`, `;`). A pattern like `Bash(find *)` does NOT match `find /path | sort` because `sort` is a separate piped command
- Specific patterns like `Bash(git *)` also fail for chained commands: `cd /path && git checkout -b branch`
- **Rule: Always use `Bash` (no parens) for blanket allow**
