# Design: `/just-ship-vps` — Autonomes Dev-Environment auf VPS

## Overview

Push-basiertes Pipeline-System auf einem VPS. Das Board hat einen "Develop" Button pro Ticket. Klick sendet einen HTTP Request an den VPS, die Pipeline läuft autonom, erstellt einen PR und setzt das Ticket auf `in_review`.

Kein Polling. Kein manuelles SSH. Alles wird von lokalem Claude Code aus eingerichtet.

## User Flow

```
User: "Richte just-ship auf meinem VPS ein."

Claude: "Ich brauche von dir 4 Dinge:"

  1. VPS IP-Adresse         → Hostinger Dashboard → VPS → IP kopieren
  2. SSH-Zugang              → Root-Passwort oder bestehender SSH-Key
  3. GitHub Token            → [Schritt-für-Schritt-Anleitung]
  4. Subdomain + A-Record   → z.B. dev.deinedomain.de → A-Record auf VPS-IP

"Wenn du mir das gibst, mache ich den Rest."

Claude richtet per SSH alles autonom ein.
```

## Architecture

```
Board (board.just-ship.io)
    │
    │  POST https://dev.userdomain.de/api/launch
    │  X-Pipeline-Key: <secret>
    │  { ticket_number: 267, project_id: "uuid" }
    │
    ▼
Caddy (HTTPS, auto Let's Encrypt)
    │
    ▼
Docker: pipeline-server (localhost:3001)
    │
    │  Auth check → project config lookup → git pull → pipeline
    │
    ▼
GitHub PR + Ticket-Status → "in_review"
```

## Phase 1: VPS Setup (`/just-ship-vps`)

### Prerequisites (4 Dinge vom User)

| # | Input | Anleitung die Claude gibt |
|---|-------|---------------------------|
| 1 | **VPS IP-Adresse** | Hostinger Dashboard → VPS Overview → IP kopieren |
| 2 | **SSH-Zugang** | Root-Passwort für initialen Zugang. Claude richtet SSH-Key-Auth ein und deaktiviert Passwort-Login danach. Falls der User schon einen SSH-Key hat, prüft Claude das vorher. |
| 3 | **GitHub Token** | github.com → Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → Generate new token → Scopes: `repo` + `workflow` → Generate → Token kopieren (wird nur einmal angezeigt) |
| 4 | **Subdomain + DNS** | Beim Domain-Provider (Hostinger, Cloudflare, Namecheap, etc.): Neuen A-Record anlegen. Name: `dev` (oder `pipeline`, etc.), Wert: VPS-IP. Dann ist z.B. `dev.deinedomain.de` erreichbar. |

### Was Claude autonom macht (per SSH)

1. **SSH-Key einrichten** — lokalen Public Key auf VPS kopieren, Passwort-Login deaktivieren
2. **System-Update** — `apt-get update && upgrade`
3. **Docker installieren** — official Docker apt repo, `docker-ce` + `docker-compose-plugin`
4. **`claude-dev` User** — `useradd -m -s /bin/bash claude-dev`, add to `docker` group
5. **Just-Ship Repo klonen** — `git clone` nach `/home/claude-dev/just-ship/`
6. **Caddyfile schreiben** — mit der Subdomain des Users
7. **Docker Image bauen + starten** — `docker compose up -d` (Caddy + Pipeline Server)
8. **Pipeline Key generieren** — zufälliger 64-Zeichen-Key, in Server-Config speichern
9. **Verifizieren** — `curl https://dev.userdomain.de/health`
10. **Board konfigurieren** — Pipeline-URL + Key im Workspace hinterlegen

### Ergebnis

VPS läuft. HTTPS aktiv. Server wartet auf Trigger vom Board. User muss nichts weiter tun.

## Phase 2: Projekt verbinden

Separater Step, pro Projekt. Wird am Ende von Phase 1 angeboten oder später ausgeführt.

### Was passiert

1. **Lokale Projekt-Config lesen** — `project.json`, `.env`-Dateien
2. **Env Vars per SSH kopieren** — nach `/home/claude-dev/.env.<projekt-slug>` (chmod 600)
3. **Repo auf dem VPS klonen** — nach `/home/claude-dev/projects/<slug>/`
4. **`setup.sh` ausführen** — just-ship Pipeline im Projekt installieren
5. **Projekt in Server-Config registrieren** — in `server-config.json` eintragen

### Server Config Format

```json
{
  "server": {
    "port": 3001,
    "pipeline_key": "<auto-generated-64-char-secret>"
  },
  "workspace": {
    "workspace_id": "<uuid>",
    "board_url": "https://board.just-ship.io",
    "api_key": "<workspace-api-key>"
  },
  "projects": {
    "aime": {
      "project_id": "<uuid-from-board>",
      "repo_url": "https://github.com/org/aime.git",
      "project_dir": "/home/claude-dev/projects/aime",
      "env_file": "/home/claude-dev/.env.aime"
    }
  }
}
```

- `server.pipeline_key` — authentifiziert **eingehende** Requests vom Board
- `workspace.api_key` — authentifiziert **ausgehende** Calls zur Board API (Ticket-Status-Updates)
- `projects[].env_file` — per-project Env Vars, werden bei Pipeline-Launch geladen

### Env Vars per Projekt

Aus dem lokalen Projekt kopiert nach `/home/claude-dev/.env.<slug>` (chmod 600):

- `ANTHROPIC_API_KEY` — Claude API Key
- Projekt-spezifische Vars (Supabase URL, Service Key, etc.)
- `GH_TOKEN` — global, in `/home/claude-dev/.env` (shared über alle Projekte)

**Runtime Env Loading:** Server liest bei Launch-Request den `env_file` Pfad aus `server-config.json`, parst die `.env.<slug>` Datei, und übergibt die Vars an `executePipeline()` via neuem `env` Parameter auf `PipelineOptions`. Jeder Pipeline-Run bekommt eine isolierte Umgebung.

## Security

| Layer | Schutz | Implementierung |
|-------|--------|-----------------|
| **Transport** | HTTPS Verschlüsselung | Caddy — automatisches Let's Encrypt Zertifikat, auto-renewal |
| **Auth** | Nur das Board darf triggern | `X-Pipeline-Key` Header mit 64-Zeichen-Secret, geprüft bei jedem Request |
| **SSH** | Key-only Login | Passwort-Login wird bei Setup deaktiviert |
| **Secrets** | API Keys geschützt | `.env` Dateien mit `chmod 600`, nur `claude-dev` kann lesen |
| **Isolation** | Container-Grenze | Docker — Pipeline läuft isoliert vom Host |

## Server Changes (server.ts)

Aktueller server.ts ist per-project (hardcoded `SUPABASE_PROJECT_ID`). Muss workspace-level werden.

### Request Flow

1. Request: `POST /api/launch` mit `{ ticket_number, project_id }`
2. **Auth:** `X-Pipeline-Key` Header prüfen gegen `server-config.json → server.pipeline_key`
3. **Lookup:** `project_id` aus Request Body → Eintrag in `server-config.json.projects` finden. `404` wenn nicht gefunden.
4. **Load Env:** `env_file` des Projekts parsen (z.B. `/home/claude-dev/.env.aime`)
5. **Execute:** `executePipeline({ projectDir, ticket, env })` — `env` Parameter ist neu, injiziert per-project Vars in den Child-Process
6. **Report:** Pipeline erstellt PR, updated Ticket-Status via Board API mit `workspace.api_key`

Note: `run.ts` macht intern `git checkout main && git pull`. Server dupliziert das nicht.

### Concurrency

- **Ein Pipeline-Run gleichzeitig** — weitere Requests werden mit `429 Too Many Requests` abgelehnt
- `409 Conflict` wenn dasselbe Ticket schon läuft
- Kein in-memory Queue (State wäre bei Restart verloren)
- Board kann retrien oder "Pipeline ist gerade beschäftigt" anzeigen
- Future: mehrere parallele Pipelines (braucht mehr VPS-Ressourcen)

### API Endpoints

```
POST /api/launch          — Pipeline für ein Ticket triggern
POST /api/answer          — Pausierte Pipeline fortsetzen (human-in-the-loop) [existing]
POST /api/ship            — PR für ein Ticket mergen [existing]
GET  /health              — Server-Status (running ticket, busy/idle)
GET  /api/status/:ticket  — Pipeline-Status für ein bestimmtes Ticket
```

Existierende Endpoints (`/api/answer`, `/api/ship`, `/api/events`) bleiben erhalten und werden für Multi-Project refactored.

### Response Examples

```json
// POST /api/launch → 202 (accepted, pipeline starting)
{ "status": "accepted", "ticket_number": 267, "project": "aime" }

// POST /api/launch → 409 (same ticket already running)
{ "status": "conflict", "ticket_number": 267, "message": "Already running" }

// POST /api/launch → 429 (another pipeline is running)
{ "status": "busy", "message": "Pipeline busy", "running_ticket": 123 }

// POST /api/launch → 404 (project not found)
{ "status": "error", "message": "Project not found for project_id: xxx" }

// GET /health → 200
{ "status": "ok", "running": { "ticket_number": 267, "started_at": "2026-03-22T14:30:00Z" } }

// GET /health → 200 (idle)
{ "status": "ok", "running": null }
```

### Pipeline Status Tracking

In-Memory State im Server:

```ts
interface PipelineState {
  running: {
    ticketNumber: number;
    projectSlug: string;
    startedAt: Date;
  } | null;
}
```

Ephemeral — bei Restart verloren. Source of Truth für Ticket-Status ist immer das Board/Supabase. Der In-Memory State dient nur für `/health`, `/api/status` und Concurrency-Gating.

## Docker

### Dockerfile (`vps/Dockerfile`)

```dockerfile
FROM node:20-slim

# System dependencies
RUN apt-get update && apt-get install -y \
    git curl jq unzip build-essential \
    && rm -rf /var/lib/apt/lists/*

# GitHub CLI
RUN curl -fsSL https://cli.github.com/packages/githubcli-archive-keyring.gpg \
    | dd of=/usr/share/keyrings/githubcli-archive-keyring.gpg \
    && echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" \
    | tee /etc/apt/sources.list.d/github-cli.list \
    && apt-get update && apt-get install -y gh \
    && rm -rf /var/lib/apt/lists/*

# Claude Code CLI
RUN npm install -g @anthropic-ai/claude-code

# Pipeline SDK
COPY pipeline/ /app/pipeline/
WORKDIR /app

# Entrypoint: configure git + gh auth, then start server
COPY vps/entrypoint.sh /app/entrypoint.sh
RUN chmod +x /app/entrypoint.sh

ENTRYPOINT ["/app/entrypoint.sh"]
CMD ["npx", "tsx", "pipeline/server.ts"]
```

### Entrypoint Script (`vps/entrypoint.sh`)

Läuft bei Container-Start vor dem Server:

```bash
#!/bin/bash
set -e

# Configure git identity
git config --global user.name "Claude Dev"
git config --global user.email "claude-dev@pipeline"
git config --global init.defaultBranch main

# Authenticate GitHub CLI with GH_TOKEN (from env)
if [ -n "$GH_TOKEN" ]; then
  echo "$GH_TOKEN" | gh auth login --with-token 2>/dev/null
  gh auth setup-git
fi

# Execute CMD
exec "$@"
```

### docker-compose.yml (`vps/docker-compose.yml`)

```yaml
services:
  caddy:
    image: caddy:2
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
      - caddy_config:/config

  pipeline-server:
    build:
      context: ..
      dockerfile: vps/Dockerfile
    restart: unless-stopped
    # Kein Port nach außen — nur Caddy kann zugreifen
    expose:
      - "3001"
    volumes:
      - /home/claude-dev/projects:/home/claude-dev/projects
      - /home/claude-dev/.just-ship:/home/claude-dev/.just-ship
    env_file:
      - /home/claude-dev/.env
    environment:
      - SERVER_CONFIG_PATH=/home/claude-dev/.just-ship/server-config.json

volumes:
  caddy_data:
  caddy_config:
```

**Build Context:** `context: ..` = Repo-Root, damit `COPY pipeline/` funktioniert. Compose muss aus `vps/` gestartet werden oder mit `-f vps/docker-compose.yml` vom Repo-Root.

### Caddyfile (`vps/Caddyfile`)

Wird bei Setup dynamisch mit der User-Domain geschrieben:

```
dev.userdomain.de {
    reverse_proxy pipeline-server:3001
}
```

3 Zeilen. Caddy holt sich automatisch ein Let's Encrypt Zertifikat.

## Board Integration

Das Board braucht einen "Develop" Button auf Tickets der:

1. `POST` an die konfigurierte Pipeline-URL des Workspaces sendet
2. `X-Pipeline-Key` Header mitsendend
3. `{ ticket_number, project_id }` im Body
4. Feedback anzeigt: "Pipeline gestartet" / "Beschäftigt" / Fehler

Board Workspace Settings:
- `pipeline_url` — die HTTPS URL (z.B. `https://dev.userdomain.de`)
- `pipeline_key` — das shared Secret

## Update Strategy

Just-Ship auf dem VPS aktualisieren:

```bash
# Via SSH (automatisiert durch Claude Code)
cd /home/claude-dev/just-ship && git pull
docker compose -f vps/docker-compose.yml build --no-cache
docker compose -f vps/docker-compose.yml up -d
```

Future: `/just-ship-vps update` Subcommand.
